Three decisions worth questioning before locking in

  1. fb_entries floating disconnected. Per-day aggregation with no FK to movies or screens. Matches the legacy POS shape but means we can never SQL-join "what movie was playing when F&B was strong". If we want that correlation, add an optional screen_id FK.
  2. movies.cinema_id as a hard FK. A movie belongs to one cinema. If you play the same film at multiple locations, you'd have one row per cinema (because share % is negotiated per-location). Alternative: shared movies + movie_screenings join. The per-cinema row is correct for Abhinaya.
  3. Three M:N joins all hang off classes. Deleting a class means detaching from screens + every price card + every serial start. RESTRICT FKs force that. The UI needs a clear "detach first" error path. is_active soft-retire is the day-to-day path; hard delete is for typos only.

Smaller gut-checks

  • openings.vals is JSONB (Partial<CumulativeRow>) — short term fine, long term could normalize.
  • serial_starts + serial_start_classes could collapse into a single denormalized table. Three tables → one. Current shape matches the in-memory AppState; either works.
  • entries.shows stays JSONB. Class IDs inside shows[].rows are loose references (validated in app code).
